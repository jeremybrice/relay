Portable Learning + Memory

Relay grows from a session-handoff tool into a handoff plus lightweight self-learning tool — so agents get smarter in every repo it lives in, not just the Forge workspace. Active, deterministic, daemon-free, local-only.

01

The big picture — one tool, two tiers

Relay already carries the daily handoff. We add a second, durable tier alongside it — knowledge that outlives the rolling 10-day window — without adding any cron, batch analyzer, or daemon. The deterministic helper still does every file operation; the agent still only authors prose.

Core — deterministic helper (relay.sh) Ephemeral — daily handoffs Durable — learned knowledge Graduated — standing instructions

TIER 1 Ephemeral handoffs

The existing system, unchanged. A 6-section handoff per session, a rolling 10-day history, an index line per day. Pruned at 10 days. Answers "what happened recently?"

TIER 2 · NEW Durable knowledge

Facts + lessons that never prune. Captured at the same save moment, loaded every session. Lessons reinforce on recurrence and graduate into the repo's instruction file. Answers "what is permanently true here, and what have I learned about working here?"

What we deliberately did not port from Forge: the SessionObserver cron, the /analyze-learning batch pipeline, confidence-score math, and time-decay. Those depend on a daemon a dogfood repo won't have. We keep the idea — capture → reinforce → graduate — and drop the machinery.

03

Data structure — files to folders

Everything new lives under the existing .session-log/ directory, so the gitignore footprint stays a single line. Green = new in this design. The tool (.relay/) is committed and travels via git; the data (.session-log/) is local-only and gitignored.

repo-root/ ├─ .relay/committed · the tool travels │ ├─ relay.sh # core helper · gains `knowledge` subcommands + subcmds │ └─ adapters/ # templates the installer wires from │ ├─ .session-log/git-ignored · local-only │ │ │ │ ── TIER 1 · ephemeral handoffs (unchanged) ────────── │ ├─ latest.md # newest single-session handoff keep │ ├─ index.md # one line per day, ≤10 days keep │ ├─ history/ │ │ ├─ 2026-06-22.md # one file per day · same-day appended │ │ └─ … # last 10 dated files · older pruned │ │ │ │ ── TIER 2 · durable knowledge (NEW · never pruned) ── │ └─ knowledge/ NEW │ ├─ index.md # DERIVED CACHE — rebuilt from files every op NEW │ ├─ facts/ # one file per fact (id = filename) NEW │ │ ├─ deploy-release-script.md │ │ └─ superseded/ # tombstones — retired / lost-conflict facts │ └─ lessons/ # one file per active lesson NEW │ ├─ schema-types-first.md │ ├─ graduated/ # moved here on graduation (+ back-pointer) │ └─ superseded/ # tombstones — retired lessons │ ├─ .gitignore # one entry already covers .session-log/ └─ CLAUDE.md / AGENTS.md # gains a marked block for graduated lessons + block <!-- relay:learned --> … <!-- /relay:learned -->

index.md is a derived cache — relay.sh regenerates it from the entry files on every locked op (never hand-edited), so it can't drift from the files. It carries every field the load path ranks on, so load reads only the index. Each entry is its own small file, keyed by id — the atomic single-file rewrite is what makes the counter-bump safe.

facts/<id>.md a durable truth

---
id: deploy-release-script
kind: fact
confirmed: 3          # reinforcement proxy
last_confirmed: 2026-06-22
ttl: none             # optional freshness window
source: history/2026-06-22.md#session-2
status: active
---
Deploys ship via scripts/release.sh —
never `npm publish` directly.

Never graduates — a fact isn't an instruction. confirmed + ttl + helper-stamped source give it dedup, freshness, and an audit trail with no math.

lessons/<id>.md a behavioral pattern

---
id: schema-types-first
kind: lesson
seen: 3   sessions: 3     # DISTINCT dates
last_seen: 2026-06-22
source: history/2026-06-22.md#session-2
status: active   graduated_to: null
---
When editing the DB schema, regenerate
types before touching call-sites…

Graduates on sessions ≥ 3 (distinct dates, not raw seen). graduated_to back-pointer lets list catch drift if its rule is later deleted.

Save flow — one reflection, three destinations

Capture piggybacks entirely on the save you already run. No new habit. At wrap-up the agent reflects once and routes that reflection into three buckets — the helper does the deterministic writing.

A durable fact. Agent runs --near first to reuse an existing id, then writes with an explicit --id. A matching id reinforces; a divergent body raises a conflict, never a silent overwrite.

Division of labor, unchanged from Relay's DNA. The agent owns the judgment — is this durable? does it match an existing lesson? what's the id? The helper owns the deterministic mechanics — append, bump, dedup, merge, lock. A mid-session "remember this" runs the same relay knowledge add via the standalone /relay-learn.

Graduation relocates, it doesn't shrink total context. A graduated lesson moves out of slot 5 and into the instruction file the harness loads natively — keeping Relay's injected block small and avoiding double-injection. But that file is uncapped, so load also nudges if the graduated region grows large, and ungraduate drains it. (Corrected from the pre-review "budget control" framing.)

Lesson lifecycle — the whole learning engine

No confidence math, no cron. Just a counter, a threshold, and a gated append. This is the entire "self-learning" mechanism.

Retirement path (new): a graduated rule that goes stale is drained by relay knowledge ungraduate <id> — idempotently removes its lines from the marked block and tombstones it. No graduated text is ever orphaned. The sessions gate (distinct dates) replaces a bare counter so a coincidence seen three times in one afternoon can't graduate like a real pattern.

gated, not automatic Why you approve graduation

Writing into CLAUDE.md/AGENTS.md is a persistent, behavior-changing edit, so per your decision-threshold rules the agent proposes it — "this lesson's been confirmed 3×, graduate it?" — and you okay it. An RELAY_AUTO_GRADUATE toggle exists for hands-free use, off by default.

retire No decay — supersede by judgment

Forge's time-decay needs a scheduler we don't have. Instead a lesson that turns out wrong is retired deliberately: relay knowledge supersede <id> flips it to superseded so it stops loading. Cutting decay is part of "lighter."

Hooks & adapters — symmetric across harnesses

Both adapters call the same relay.sh; only the trigger wiring and the context-injection field differ. The new design adds exactly one capture instruction and one standalone command per adapter — the load hook is untouched in wiring, it just emits more.

Why the fix lives in the shared core, not the adapters. A single append inside cmd_load flows to both harnesses for free and keeps them from drifting — the same boundary discipline that produced the full-day-load decision. Adapters stay thin: trigger wiring + injection field only.
09

Command surface — all deterministic

Every write is a helper subcommand the agent calls. The agent never edits the knowledge files directly — that's what makes graduation idempotent and the counters trustworthy.

# ── capture (piggybacks on save, or standalone via /relay-learn) ──
relay knowledge add --fact   --near "…"          # show top-3 existing ids, write nothing
relay knowledge add --fact   --id deploy-release-script "…"
        # new → confirmed:1 · matching → confirm · divergent body → CONFLICT (not overwrite)
relay knowledge add --lesson --id schema-types-first "…"
        # new → seen:1 · re-asserted → bump seen + (sessions on a new day)

# ── resolve / retire ──
relay knowledge resolve    deploy-release-script   # pick a side of a fact conflict
relay knowledge supersede  <id>                    # tombstone an active entry
relay knowledge prune                              # propose expired/stale facts (gated)

# ── graduate a reinforced lesson into the instruction file (gated) + drain it ──
relay knowledge graduate   schema-types-first    # idempotent id-keyed block write
relay knowledge ungraduate schema-types-first    # retire a stale graduated rule

# ── audit / inspect ──
relay knowledge why <id>     # cat the entry + the handoff it came from (provenance)
relay knowledge list          # current state + conflict/drift flags
relay knowledge export        # concat active entries → one shareable file (gated)

# ── load now emits Facts + active Lessons; truncation is NEVER silent ──
relay load [--format codex]

Local-only, for now. The whole knowledge tier is gitignored like the handoffs. 11

Boundary & non-goals

What keeps this "lighter and active" instead of a second Forge.

✓ In scope — v1 of learning

  • Durable facts + behavioral lessons, captured at save
  • Reinforcement-lite — a seen counter, no confidence scores
  • Gated graduation into the instruction file at threshold
  • Separate-budget load so the handoff is never squeezed
  • Standalone capture via /relay-learn
  • Extended zero-dependency test suite

✗ Out of scope — deliberately cut

  • No cron, background capture, or batch analyzer
  • No time-decay (no scheduler to run it)
  • No confidence-score math — just the counter + threshold
  • No committed / shared store yet (config switch reserved)
  • No cross-repo knowledge aggregation
  • No auto-graduation by default (toggle exists, off)
The boundary invariant. Knowledge is just markdown, fully readable with no adapter; relay.sh owns every write, merge, and counter; the agent owns judgment and prose; adapters stay thin. Relay grows a new capability without losing the deterministic DNA that made it portable.
12

Hardened after a 3-agent design review

A Cognitive Forge review (challenger / explorer / decomposer) found the v1 draft's determinism was broken at one seam — the id/slug contract — with index.md consistency as the second concentrated risk and three holes in graduation. The core thread: the file write was deterministic, but the create-vs-merge-vs-overwrite outcome rode on an unverified agent slug. All six critical issues are now folded in.

#Issue the review surfacedFix folded into this design
R1Fact id contract undefined & self-contradictory (--fact had no id, but merge keyed on id)Facts take an explicit --id; deterministic slug-normalization rules pinned
R2Confirm-and-merge silently overwrote on an id collision — data loss on the happy path--near dedup before write; id match + divergent body → conflict block + resolve, never overwrite
R3Facts cap reintroduced the silent partial continuity the full-day-load spec was written to killTruncation is never silent — load emits "N of M shown … may include load-bearing truths"
R4index.md was a single point of failure that capped load hides when it driftsindex.md is a derived cache, full-rebuilt from entry files every locked op — drift made structurally impossible
R5Graduation: uncapped surface, no retirement path, leaks local learning into a committed fileungraduate drains it; "budget control" claim corrected; the local→committed leak is named & gated
R6Bare seen ≥ 3 couldn't tell a coincidence from a patternGate is now sessions ≥ 3 — distinct dates, fed by helper-stamped provenance

enh Provenance

Helper stamps source: history/<date>.md#session-N at zero agent cost — unlocks contradiction-handling, the sessions gate, and relay knowledge why.

enh confirmed + TTL

A fact reinforcement counter (free confidence proxy) and an optional freshness window revalidated lazily at load — staleness needs no scheduler.

enh Conflict markers

Git's "never auto-resolve, always surface" discipline for contradictory facts: both sides kept, flagged at load, resolve picks one.

Verdict the three agents shared: the architecture is sound and buildable in deterministic bash 3.2 — the fragility was concentrated in two contracts (fact id + index.md), not pervasive. Both are now pinned. Full review: 2026-06-22-relay-learning-design-review.md
